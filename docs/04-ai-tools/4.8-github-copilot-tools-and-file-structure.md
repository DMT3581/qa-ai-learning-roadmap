# GitHub Copilot Tools and File Structure

> **⚠ Fast-moving feature.** Copilot's customization files and modes change often (new file types, renamed features, preview flags). Treat specifics here as a **June 2026 snapshot** and verify current file names and behaviour in GitHub's official docs before relying on them.

## In one line

Copilot reads special files in your repo to learn your project and to shape how it behaves — and the single highest-leverage one is `.github/copilot-instructions.md`, which teaches Copilot your conventions once so you stop repeating them in every prompt.

## Why this matters for QA

If you (or your automation colleagues) use Copilot in a repo, a few well-placed files change it from a generic assistant into one that knows your stack, your test framework, and your house style. For QA that's directly useful: put your test conventions in an instructions file and Copilot's generated tests start matching your patterns instead of inventing their own. Knowing which file does what — and which customization mechanism to reach for — is the difference between fighting Copilot and having it fit your project.

## Core content

### The file structure: how Copilot learns your project

Copilot looks for specific files in your repository and folds them into its context. The main ones:

| File / location | What it does |
|---|---|
| `.github/copilot-instructions.md` | Repo-wide custom instructions, auto-applied to **every** chat/agent request. The single most important file — project summary, tech stack, structure, conventions. |
| `.github/instructions/*.instructions.md` | Path-specific instructions with an `applyTo` glob in the frontmatter — applied only when Copilot works on matching files (e.g., rules just for your test files). |
| `.github/prompts/*.prompt.md` | Reusable, on-demand task templates invoked with `/name` — "generate a test", "review this PR". Your team's recipes. |
| `.github/agents/*.agent.md` | Custom agents (formerly chat modes): named personas with a defined tool set, model, and boundaries for a whole session. |
| `.github/skills/<name>/SKILL.md` | Folder-based capability packages (instructions plus scripts/templates), loaded on demand. Portable across Copilot tools. |
| `AGENTS.md` (root or nested) | Cross-tool agent instructions in an open format shared with Claude Code, Cursor, and Gemini (also `CLAUDE.md` / `GEMINI.md`). |

Precedence when several apply: personal → repository → organization (all relevant ones are still provided, so avoid conflicting rules).

### Which mechanism for which need

A useful decision framework:

- Rule applied to **every** request, no one has to remember? → **Instructions** (`copilot-instructions.md` / `*.instructions.md`).
- A **named, repeatable task** invoked on demand? → **Prompt file**.
- A **specialist persona** with a limited tool set or a multi-step handoff? → **Custom agent**.
- A **reusable runbook** with bundled scripts and templates? → **Skill**.
- Needs **live external data or actions**? → add **MCP** (see *MCP in Practice*).
- Must be enforced **deterministically**, no chance the model ignores it? → add a **Hook**.

The first four are "pick one"; MCP and Hooks are layers you add on top.

### Copilot's tools (modes and tool-calls)

"Tools" means two related things. The **modes** you interact through — inline completions and next-edit suggestions, Chat (Ask), Plan, Edits across files, Agent mode, code review, the CLI, and the cloud agent. And the **tool-calls** the agent can make under the hood — reading and editing files, searching, running terminal commands, web access — which are exactly the harness tools from *Harness Tools*. A custom agent can restrict which of these are available (for example, a review agent that can read and lint but never edit).

### Two caveats worth knowing

- **Instructions don't affect inline completions.** The customization files shape Chat, Plan, and Agent interactions — not the gray-text autocomplete as you type.
- **Keep instructions concise.** Copilot is probabilistic; a focused instruction file "tilts the scales" toward what you want. Bloated instructions dilute rather than help.

## Examples

- **`.github/copilot-instructions.md`** stating "This is a Playwright + TypeScript test repo; follow the Page Object pattern; name tests `feature.spec.ts`" → generated tests match your conventions without you re-explaining each time.
- **A `generate-test.prompt.md`** prompt file → `/generate-test` scaffolds a test in your house format on demand (your prompt library, in the repo — see *Personal Prompt Library*).
- **A "Reviewer" custom agent** with read-and-lint tools but no edit access → a critical review pass that can't accidentally change your code (see *AI as a Peer Reviewer* and *Human-in-the-Loop Automation*).

## Try it yourself *(optional)*

Add a short `.github/copilot-instructions.md` to a test repo — a few lines on the stack, the test framework, and your naming and structure conventions — then ask Copilot to generate a test and see how much closer it lands to your style. That one file is usually the highest-return customization you can make.

## Common pitfalls

- **No instructions file.** You re-explain your project in every prompt.
- **Bloated instructions.** Long, unfocused files dilute the signal.
- **Conflicting instructions** across files — resolution is non-deterministic.
- **Expecting instructions to change inline completions.** They only affect chat/agent.
- **Reaching for the wrong mechanism** — a one-off task doesn't need a custom agent; a persistent rule shouldn't be a prompt file.

## Key takeaways

- Copilot reads repo files to learn your project; `.github/copilot-instructions.md` is the key one.
- Instructions (always-on), prompt files (on-demand), custom agents (personas), and skills (runbooks) each serve a different need.
- MCP and Hooks are additive layers for live data and hard enforcement.
- Instructions shape chat and agent modes, not inline completions — and concise beats long.
