# Understanding AI Tool Categories and Capabilities

## In one line

AI tools aren't interchangeable — placing any tool on two axes (what *kind* of tool it is, and what it can actually *do*) tells you what to expect before you ever pick one.

## Why this matters for QA

You'll meet several different kinds of AI tool: a chat assistant, an AI button inside Word or Jira, a coding copilot in your IDE, and increasingly agents that take actions for you. They are not the same, and treating them as interchangeable leads to wrong expectations — a chatbot won't run your test suite; an agent might — and wasted effort. Knowing the categories and what gives a tool its power lets you reach for the right kind of tool for a task, and sets up the next chapters on choosing and comparing specific tools.

## Core content

### Axis 1 — what kind of tool is it

- **General-purpose chat assistants.** Text in, text out; the output stays in the chat and you copy it out and act on it. Best for thinking, drafting, and analysis. *(ChatGPT, Claude, Gemini.)*
- **Embedded / in-app copilots.** AI built into a tool you already use, working in that app's context. *(Microsoft 365 Copilot in Word/Excel/Teams; copilots in Jira, Azure DevOps, and test-management tools.)*
- **Coding assistants.** AI inside the developer's editor — inline completion, chat, and agent modes. *(GitHub Copilot, Cursor.)*
- **Autonomous agents.** Tools that take actions across apps and systems and run multi-step workflows with less supervision. The defining 2026 shift is from "tells you what to do" to "does it for you." *(Computer-use agents, Playwright-MCP-driven flows, agentic coding tools.)*
- **Specialized / single-purpose tools.** Do one thing very well but can't cross app boundaries. *(Meeting transcription, image generation, research/search, dedicated AI testing tools.)*

Specific products change fast — the **categories** are the durable part.

### Axis 2 — what can it actually do (capabilities)

The same category can be weak or powerful depending on its capabilities:

- **Underlying model (and version)** — the engine; a newer or stronger model is more capable (see *Model Versions*).
- **Modality** — text only, or also code, images (can it *see* a screenshot?), audio, video.
- **Tool access** — web search (recency), code execution (real math, running code), file/document handling, browsing. This is the big one: a model *with tools* overcomes many of the raw limits from *Practical Limitations*.
- **Context window** — how much it can take in at once (see *Context Windows*).
- **Memory** — whether it persists anything across sessions.
- **Integration** — what systems it connects to (your repo, M365, Jira).
- **Deployment & governance tier** — consumer vs. enterprise, cloud vs. on-prem; this decides what's safe to put into it (see *Confidentiality and Sensitive Data*).

### Put the two axes together

To size up any AI tool, ask: *what category is it, and what capabilities does it have?* A general chatbot with no tools can't fetch today's data or run your code; the same chatbot *with* web search and code execution can. An agent can act, but needs guardrails. That lens is what you carry into choosing a tool for a task and comparing specific products.

## Examples

- Brainstorm test ideas or draft cases → a **general chat assistant**.
- Write automation code in your IDE → a **coding assistant**.
- Summarize a Teams meeting or analyze a results spreadsheet → an **embedded copilot**.
- Actually drive the browser and execute steps → an **agent** (your Layer 2 Playwright-MCP territory).
- "Can it look at this screenshot of the failing UI?" depends on **modality**; "does it know the newest framework version?" depends on **tool access** (web search).

## Try it yourself *(optional)*

Take one task you'd want AI help with this week. Name which category fits, and which capabilities it needs — does it need tools, multimodality, or integration with your systems? If your go-to tool lacks them, that's exactly why it's been underwhelming on that task.

## Common pitfalls

- **Treating all AI tools as interchangeable.** Category and capability both matter.
- **Expecting a chatbot to take actions.** It produces text; you execute.
- **Ignoring capabilities.** A tool-less model can't fetch recent data or run code, whatever its category.
- **Ignoring the governance tier.** A consumer tool isn't automatically safe for work content.
- **Assuming "agent" means reliable autonomy.** Agents still need oversight (see the Agents chapters and *AI Does Not Replace Human Judgment*).

## Key takeaways

- Place any tool on two axes: its category (interface/behaviour) and its capabilities.
- Categories: chat assistants, embedded copilots, coding assistants, agents, specialized tools.
- Capabilities — model, modality, tool access, context, memory, integration, tier — decide real power.
- Tools plus capabilities, not category alone, determine what a tool can do for a task.
