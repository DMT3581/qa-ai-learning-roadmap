# Harness Tools: What a Harness Is and Why It Matters

## In one line

A harness is all the software wrapped around an LLM — the loop, the tools, the memory, the guardrails — that turns a text-in/text-out model into something that can actually *do* things; as the saying goes, "if you're not the model, you're the harness."

## Why this matters for QA

The same model can be nearly useless or genuinely powerful depending on the harness around it — which is why Claude in a chat window just talks, while Claude inside Claude Code can read your files, run your tests, and commit code. Once you see that "the tool" is a model *plus* a harness, a lot clarifies: why the same model behaves differently across tools, why "the AI is bad at this" is often really "this harness is bad at this," and why your Layer 2 work — Playwright MCP plus the Claude API plus the orchestration around them — is essentially building a test harness around a model.

## Core content

### The model alone doesn't do much

An LLM is text in, text out (see *What Is a Large Language Model*). By itself it has no memory between calls, no loop, and — importantly — it can *express* the intent to use a tool but can't actually execute anything. It answers one prompt and stops.

### The harness is everything around the model

A harness is the software scaffolding wrapped around the model that lets it act: the orchestration **loop**, the **tools** and the code that actually runs them, **context and memory management**, the **system prompt**, response **parsing**, **error handling**, **permissions and guardrails**, the **sandbox**, and session **history**. The widely quoted rule of thumb is "if you're not the model, you're the harness," which gives the tidy formula **Agent = Model + Harness** (see *What Is an AI Agent*).

Anthropic frames its own tool exactly this way: Claude Code is "the agentic harness around Claude" — it provides the tools, context management, and execution environment that turn a language model into a capable coding agent.

### Same model, different harness, different powers

This is the key insight. The *same* Claude that, in the chat web app, can only produce replies can, inside Claude Code, read and edit your source, run tests, and commit — because the harness is heavier. A minimal harness gives you a chatbot; a hefty one gives you an agent that acts on the filesystem. So a tool's capability comes from its harness as much as from its model (see *Understanding AI Tool Categories and Capabilities*).

A striking illustration: when people picked apart a real coding harness, the software turned out to be overwhelmingly infrastructure, with only a tiny fraction being the actual "AI logic." Most of what makes an agent work is the harness, not the model call.

### Harnesses are getting thinner — and are coupled to the model

As models get more capable, harnesses tend to get *thinner*: scaffolding that a weaker model needed can be stripped away. There's also tight coupling — modern models are often trained alongside a specific harness, so swapping harness pieces can change performance. Practically, the harness is a real, engineered part of the product, not an afterthought.

### Why this matters for you

- **Tool choice is harness choice.** Picking Claude Code vs. a chat window vs. Copilot means choosing a harness as much as a model (see *How to Choose the Right AI Tool*).
- **"The AI is bad at this" is often "the harness is bad at this."** A capable model in a weak harness underperforms; the fix may be a better tool, not a better model.
- **Your Layer 2 is a harness.** Playwright MCP + the Claude API + your orchestration, guardrails, and reporting *is* a test harness built around a model. Understanding the concept is understanding what you're building.

## Examples

- **claude.ai can't run your tests; Claude Code can** — same model, different harness.
- **An agent that aced a demo but flops in CI** — usually the harness (tools, loop, verification), not the model.
- **Layer 2** — the framework you assemble around the model to drive and check tests is, by definition, a harness.

## Try it yourself *(optional)*

Compare the same model in two tools — a plain chat window and a coding or agent tool — on a task that needs file or tool access. The gap you feel between "it can only tell me" and "it can actually do it" is the harness.

## Common pitfalls

- **Conflating "the model" with "the tool."** The harness is a big part of the tool.
- **Blaming the model** for what's really a harness limitation — or the reverse.
- **Assuming a great model guarantees a great agent.** The harness has to be good too.
- **Expecting a chat-window harness** to take actions it was never built for.

## Key takeaways

- A harness is all the software around an LLM that lets it act; agent = model + harness.
- The same model is limited or powerful depending on how heavy its harness is.
- Much of an agent tool is harness infrastructure, not the model call itself.
- Choosing a tool means choosing a harness — and your Layer 2 stack is one you're building.
