# Model Versions and Why They Matter

## In one line

The model behind a tool isn't fixed — versions differ in what they know, cost, and behaviour, and they change and retire over time — so when a result needs to be repeatable, the model version is part of the environment.

## Why this matters for QA

Reproducibility is a QA instinct, and the model version is part of the "environment" in the same way a browser or OS version is part of a test setup. A prompt that reliably produced good output last quarter can behave differently after a silent update — that's a regression in a dependency, not magic. Knowing this lets you record what you used, re-test after changes, and stop treating "ChatGPT" or "Claude" as one fixed thing.

## Core content

### "A model" isn't one fixed thing

Behind a product name sit multiple models, tiers, and versions, and what answers you depends on your tier, your settings, and often an automatic router. So "I used ChatGPT" doesn't actually pin down what responded.

### What differs between versions and tiers

Newer or bigger usually means more capable, but versions also differ in:

- **Knowledge cutoff** — what recent things it knows (see *Practical Limitations*).
- **Cost and speed.**
- **Context window** — how much it can take in.
- **Tool access** — web search, code execution, and so on.
- **Behaviour and "personality"** — tone, verbosity, how literally it follows instructions.
- **Safety guardrails** — sometimes stricter or looser.

A tier — Opus/Sonnet/Haiku, Pro/Flash/Lite, Instant/Thinking — is a deliberate capability-vs-cost-vs-speed trade. Picking the right one is part of choosing the tool (see *How to Choose the Right AI Tool*).

### Same prompt, different version → different output

A prompt tuned for one model can behave differently on another, or after an update — sometimes better, sometimes worse. Newer is not always better *for your specific task*: a model can improve on average while quietly regressing on the exact thing you rely on. This is why you re-test and date your prompts (see *Prompt Iteration* and *Personal Prompt Library*).

### Models change and retire — sometimes silently

Providers update hosted models, occasionally without changing the version name, which can quietly shift behaviour — a "silent regression," where a workflow that was steady degrades with no change on your side. Models are also deprecated and retired on a schedule, so an integration tied to an old one can break. Providers give notice; it's worth watching for.

### When it matters: record and re-test

- If a result must be **reproducible or shared**, note the model/version — like recording a browser version. Put bluntly: if a result can't be replayed, it's a screenshot, not evidence.
- **Re-test** your key prompts and workflows after a model update — regression testing for prompts. A "canary" prompt whose expected output you know will quickly reveal a behaviour shift.
- **For anything built on the API (Layer 2):** pin an immutable model version rather than a moving alias, and test the replacement before an old version retires. This is just standard dependency management.

### A note on "Auto" / router modes

Convenient, but they pick a model for you to optimise speed and cost — which means less control and less reproducibility. For work that matters or must be repeatable, select the model explicitly.

## Examples

- A prompt that reliably generated good test cases last quarter now behaves differently → the model was probably updated; re-test and re-tune rather than assuming it "broke."
- You share a prompt and a colleague on a different tier gets different output → note the model so the comparison is fair.
- "Auto" gave a fast, shallow answer on a hard task → it routed to a lighter model; pick the stronger one explicitly.
- *(Layer 2)* A Playwright + Claude API pipeline → pin the model ID, and when it's slated to retire, test the replacement before switching.

## Try it yourself *(optional)*

Run one library prompt on two tiers — a fast model and a "thinking" model — and compare quality and speed. The gap is exactly why recording which model you used is worth the two seconds.

## Common pitfalls

- **Treating "ChatGPT/Claude/Gemini" as one fixed thing.** It isn't.
- **Not recording the model** behind a result you'll need to reproduce.
- **Assuming an update can't change behaviour** — or that newer is always better for your task.
- **Letting "Auto" choose** for high-stakes or repeatable work.
- **Assuming one model's knowledge cutoff applies to another.**
- *(Layer 2)* **Using a moving alias instead of a pinned version; ignoring retirement notices.**

## Key takeaways

- Versions differ in knowledge, cost, behaviour, and limits — and they change over time.
- Updates can silently shift behaviour; newer isn't always better for your task.
- Record the model when a result must be repeatable, and re-test prompts after updates.
- When building on the API, pin a version and test before migrating.
