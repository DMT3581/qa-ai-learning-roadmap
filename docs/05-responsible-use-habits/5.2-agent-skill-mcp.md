# Agent, Skill and MCP Fundamentals

## In one line

Three words you'll hear together constantly: an **agent** is the worker that acts in a loop, a **skill** is packaged know-how that teaches it *how* to do a task, and **MCP** is the standard connection that gives it *access* to external tools and data — they're complementary layers, not competitors.

### Why this matters for QA

These three terms show up everywhere in agentic tooling, and they're easy to blur together because all three "extend" what an AI can do. Getting the mental model straight saves a lot of confusion — and it maps directly onto your Layer 2 world: an agent drives the work, MCP connects it to your systems (a browser via Playwright MCP, your repo, Jira), and a skill can encode your team's procedure so the agent does things *your* way every time. You've even met skills already: the `SKILL.md` files that let some AI tools reliably create Word docs or PDFs are exactly this.

## Core content

### Agent — the worker

An agent is a model using tools in a loop to pursue a goal, deciding its own next step (covered in *What Is an AI Agent*). It's the thing that actually *does* the work — perceive, decide, act, observe, repeat. The other two concepts exist to make the agent more capable.

### Skill — the *how* (procedural knowledge)

A skill is a reusable package of know-how that teaches an agent how to perform a specific task or workflow. In practice it's a folder with a **`SKILL.md`** file — YAML frontmatter (a name and a description) plus Markdown instructions — and optionally bundled scripts, references, and templates. Think of it as an onboarding guide for a new hire: you write the procedure down once, and the agent follows it without you re-explaining.

Its defining design principle is **progressive disclosure**: at startup the agent loads only each skill's name and description (a few tokens), and reads the full instructions *only when the task matches*. That's why you can have dozens of skills installed without clogging the context window. Skills are also an **open, portable standard** now — the same `SKILL.md` works across Claude, Copilot, and other tools — so one person's skill can become the team's.

### MCP — the *what* (access)

MCP (Model Context Protocol) is the open standard that connects an agent to external tools and data — the "USB-C for AI" (covered in *MCP in Practice*). It exposes capabilities the agent can call: query a database, create a GitHub issue, drive a browser, search the web. MCP gives the agent *access*; it does not, by itself, tell the agent how to use that access well for your specific context.

### How they fit together

The clean way to hold it: **MCP gives access, skills give know-how, the agent does the work.** Anthropic puts it plainly — MCP connects the agent to external services and data, while skills provide the procedural knowledge for how to complete a task; you use both together.

A chef analogy makes it stick: **MCP** is the stocked pantry and the kitchen appliances (the ingredients and tools — the *what*); a **skill** is the chef's recipe book and technique (the *how*); the **agent** is the chef, combining them to actually cook. Having the appliances doesn't mean you know the recipe, and knowing the recipe does nothing without ingredients — which is why the two are complementary, not rival.

One practical difference worth knowing: MCP tool definitions are generally *always loaded* into context (so a handful of heavy MCP servers can crowd it and hurt accuracy), whereas skills *load on demand*. If you find yourself with far more MCP servers than skills, you may be paying a context tax for capability you could express more cheaply as a skill.

### When to reach for each

- Need the agent to **do work** toward a goal it can't script? → an **agent**.
- Need it to **follow your procedure** the same way every time (test conventions, bug-report format, a deployment runbook)? → a **skill**.
- Need it to **reach an external system** (repo, Jira, browser, database)? → **MCP**.

## Examples

- **MCP:** Playwright MCP connects the agent to a real browser so it can navigate and click (access).
- **Skill:** a `write-test-cases` skill encoding your team's Playwright + Page-Object conventions, so any agent generates tests your way (know-how).
- **Agent:** the loop that, given "add coverage for the login flow," uses the MCP browser access and the test-writing skill to actually produce and check the tests (the worker).

## Try it yourself *(optional)*

Look at a `SKILL.md` file (Anthropic publishes open ones, e.g., a PDF or document skill). Notice the shape: a name and description up front, then step-by-step instructions, maybe a script. That's the whole idea — captured procedure the agent loads only when it's relevant. Then contrast it with an MCP server, which is a *connection* to a system. The difference between the two is the whole lesson.

## Common pitfalls

- **Treating the three as competitors.** They're complementary layers.
- **Reaching for MCP when you needed a skill** (you wanted a procedure, not a connection) — or vice versa.
- **Over-loading MCP servers** and paying a context tax; a skill may be cheaper.
- **Confusing a skill with a one-off prompt.** A skill is a persistent, reusable method; a prompt is live intent for one task (see *Personal Prompt Library*).

## Key takeaways

- Agent = the worker acting in a loop; skill = the *how*; MCP = the *what/access*.
- Skills are `SKILL.md` folders that load on demand (progressive disclosure) and are now an open standard.
- MCP connects the agent to external tools and data; it doesn't supply the procedure.
- Use them together: MCP for access, skills for method, the agent to run the loop.
