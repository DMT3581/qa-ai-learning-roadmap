## In one line

Token efficiency isn't about writing the shortest possible prompt — it's about maximizing signal per token: cut the filler and irrelevant context, constrain the output, and you usually save money, stay within limits, *and* get a better answer.

## Why this matters for QA

Tokens cost money and take up the context budget — and there's a bonus: a leaner, clearer prompt often produces a *better* answer, because there's less noise to distract the model. Two things sharpen this for you: premium and API usage is metered (Copilot premium requests, and Layer 2 API costs), and working in Romanian uses more tokens than English for the same content (from *Tokens Explained*). The key nuance to hold onto: efficient does not mean terse. Stripping out necessary context to save tokens backfires — you get a worse answer and re-prompt, which costs more overall.

## Learning objectives

After this chapter you will be able to:

- Explain why token efficiency matters — cost, limits, and quality.
- Apply practical strategies to trim tokens without losing signal.
- Avoid over-compressing at the expense of the result.

## Core content

### Why it matters (three payoffs)

- **Cost.** You pay per token, both input and output — and output tokens are usually the pricier half.
- **Context budget.** Every token spent on filler is a token unavailable for the actual work, in a finite window (see *Context Windows and Memory Limitations*).
- **Quality.** A leaner prompt has less noise, so the model focuses better — concise and clear often beats verbose.

### The core principle: signal per token, not fewest tokens

This is the nuance that matters. Efficiency means removing what doesn't help — filler, redundancy, irrelevant context — while keeping every bit of *necessary* context and instruction. Over-compress and you under-specify, get a worse answer, and re-prompt, which costs more than you saved. The goal is the smallest *high-signal* prompt (the same idea as *Context Engineering*), not the shortest possible one.

### Practical strategies

- **Cut filler and pleasantries.** "Could you possibly give me a detailed explanation of…" → "Explain…". Say things once; don't repeat instructions.
- **Include only relevant context.** The single biggest lever: paste the relevant slice, not the whole document (see *Context Engineering*).
- **Constrain the output length.** You pay for output too, and models default to verbose. "In 3 bullets," "under 100 words," "just the table" — ask for exactly what you need.
- **Prefer lean structure.** A table or bullets convey more per token than rambling prose. One caveat: JSON is surprisingly token-heavy because of its punctuation — use it when you genuinely need it, not by default.
- **Compact long conversations.** Summarize older turns, or restart, rather than dragging the whole history along (see *Prompt vs. Conversation*).
- **Reuse templates.** A saved, tuned prompt is efficient by construction (see *Personal Prompt Library*).

### A useful habit: measure

If cost or limits matter, paste your prompt into a token counter and compare a trimmed version side by side. You'll see where the tokens actually go — often it's context you didn't need.

## Examples

- **Trimming a pasted spec** to just the relevant sections keeps you under the context limit *and* sharpens the analysis.
- **Asking for "the top 5 issues as a table"** instead of "review this thoroughly" cuts output tokens and gives you something usable straight away.
- **A Romanian prompt** costs more tokens than the same content in English, so efficiency pays off more (see *Tokens Explained*).

## Try it yourself *(optional)*

Take a prompt you use often. Remove the filler and any irrelevant context, add an output-length constraint, and compare both the result and the token count. The usual outcome: fewer tokens, and an answer that's just as good or better.

## Common pitfalls

- **Over-compressing.** Cutting necessary context or constraints to save tokens hurts the result.
- **Forgetting output tokens.** An unconstrained, verbose response is the silent cost.
- **Repeating instructions** or re-pasting content already in context.
- **Defaulting to JSON** when a table or bullets would be leaner.
- **Dragging full conversation history** along turn after turn.

## Key takeaways

- Token efficiency saves cost, protects the context budget, and often improves quality.
- Aim for the smallest high-signal prompt — remove filler, keep necessary context.
- Constrain output length and prefer lean structure; output tokens are the pricier half.
- Compact long chats, reuse templates, and measure when cost or limits matter.
