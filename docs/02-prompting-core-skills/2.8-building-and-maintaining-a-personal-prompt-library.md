## In one line

A prompt that worked took several rounds to get right and encodes your context — save it as a reusable template instead of rebuilding it (worse) next week.

## Why this matters for QA

You'll land on prompts that reliably do useful things — generate test cases in your exact format, turn rough notes into a clean bug report, review a test plan critically. Each one took iteration to get right and quietly encodes your app, your house style, your format. If it stays buried in chat history, you rewrite it from scratch next time, usually worse because you've forgotten the phrasing that made it work. A small personal library of your best prompts turns one-time effort into a repeatable asset — and it's the seed for sharing good prompts with the team later.

## Learning objectives

After this chapter you will be able to:

- Decide which prompts are worth saving.
- Store prompts as reusable templates with the right metadata.
- Organize and maintain a library so you actually find — and trust — your prompts.

## Core content

### What's worth saving (the reuse test)

Save a prompt if you expect to use it again; skip one-off exploratory queries. The keepers tend to fall into three types:

- **Workflow prompts** — recurring tasks: test-case generation, bug-report polish, weekly test summaries.
- **Technique prompts** — a good few-shot, role, or structured pattern that's portable across tasks.
- **Domain prompts** — ones that encode your application, standards, or house style.

### Make it a template, not a snapshot

The most useful habit: replace specific values with placeholders — `<REQUIREMENT>`, `<FEATURE>`, `<PASTE_SANITIZED_CODE>` — so you swap inputs instead of rewriting the prompt each time. Placeholders do double duty: they keep the prompt reusable and keep secrets and real data out of a saved file, so you never hard-code sensitive content (see Confidentiality and Sensitive Data).

### What each entry needs

A prompt without documentation is a recipe with no instructions. A minimal useful record:

- a clear, searchable name ("Generate test cases from a user story", not "Prompt 7");
- the prompt body with placeholders;
- one line on what it does / when to use it;
- the inputs it needs;
- an example of good output;
- a last-tested date — because behaviour shifts when models update (see Prompt Iteration and Model Versions).

Add a short version note when you improve it ("v2 — added table format").

### Organize so you can find it

Start simple: 3–5 top-level categories (by task or by what you're doing), plus a few tags for cross-cutting attributes (model, technique). Pick one place and make it the default destination — scattered prompts are why people rewrite instead of search. Don't spend a week designing the perfect system; capture prompts first, refine the structure later.

### Maintain it

A library rots without upkeep. Periodically archive prompts you haven't used in months, merge near-duplicate variants, and re-test your key prompts after a model update. Mark a prompt deprecated rather than deleting it.

### From personal to team

Your personal library is the seed. The prompts that reliably work are exactly the ones worth sharing — which is a team practice with its own chapter (see Documenting and Sharing Effective Prompts and Building Team Knowledge Around AI Usage). Keep experiments personal; promote the proven ones.

### Example — a library entry

```
Name:        Generate test cases from a user story
Use case:    Turn a story + rules into a structured case table
Inputs:      <USER_STORY>, <RULES>, <TEST_DESIGN_TECHNIQUE>
Prompt:      Using <TEST_DESIGN_TECHNIQUE>, generate test cases for:
             Story: <USER_STORY>
             Rules: <RULES>
             Output a table: ID, technique, steps, expected result.
             List assumptions separately.
Output ex.:  [link or paste a good run]
Last tested: 2026-06-10
```

### Try it yourself (optional)

Take the last prompt that worked well for you, swap its specific values for placeholders, add a one-line use case and a last-tested date, and save it somewhere you'll look. That's your library started — the next reuse already pays for it.

## Common pitfalls

- Letting good prompts die in chat history. Capture them immediately.
- Hard-coding values — or sensitive data. Use placeholders; never store secrets.
- Generic names you can't search for. Name with the words you'd actually type.
- Over-engineering the system. Capturing beats organizing; structure can come later.
- Never re-testing after model updates. Behaviour drifts; date your prompts.
- Scattering prompts across many places. One source of truth, or you'll rewrite.

## Key takeaways

- Save prompts you'll reuse: workflow, technique, and domain prompts.
- Store them as templates with placeholders, never as one-offs with hard-coded data.
- Give each a searchable name, a use case, an example, and a last-tested date.
- Keep one home, prune regularly, and promote the best ones to the team.
