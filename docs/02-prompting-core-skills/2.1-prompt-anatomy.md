## In one line

A good prompt is a mini-specification, built from a handful of parts — task, context, input, format, constraints, and optionally a role and examples — most of which you already know how to write, because they're the same things a good test spec has.

## Why this matters for QA

LLMs don't read minds; a vague prompt gets a vague result. But you already know how to write a clear specification — that's the job — and a prompt is just a spec written for the model. Knowing the building blocks does two things: it lets you assemble a strong prompt on purpose instead of by trial and error, and it lets you diagnose a weak one ("the output's in the wrong format" → you left out the format part). This is the foundation the rest of the prompting chapters build on.

## Learning objectives

After this chapter you will be able to:

- Name the building blocks of an effective prompt.
- Assemble a prompt deliberately from the parts a task needs.
- Diagnose a weak prompt by spotting the missing part.

## Core content

### A prompt is a mini-spec

Treat a prompt not as a question you toss out, but as a small specification of what you want. The clearer the spec, the closer the output. You don't always need every part — a simple ask might be just an instruction plus input, while a complex one uses them all.

### The building blocks

- **Task / instruction** — the one part you always need. The action verb: *what* to do, clearly and specifically ("summarize," "generate," "review," "convert").
- **Context** — background the model needs and couldn't know: the project, the rules, the situation (see *Context Engineering*).
- **Input / data** — the actual thing to work on: the text, code, or requirement.
- **Output format** — how the answer should be shaped: a table with named columns, JSON, a numbered list, a length limit (see *Writing Clear Constraints*).
- **Constraints** — the boundaries: scope, tone, what to include or leave out.
- **Role / persona** *(optional)* — who the model should act as, mainly to set tone and audience rather than to boost accuracy (see *Advanced Prompting Techniques*).
- **Examples** *(optional)* — a sample of what "good" looks like (few-shot), the strongest way to lock in a format (see *Advanced Prompting Techniques*).

### Order and placement

The arrangement matters. Put context before the task, and it often helps to place the key instruction near the end so it stays fresh. Separate the parts clearly — with labels or sections — so the model can tell instructions from data.

### A worked example

**Weak:** "Review my test plan."

**Assembled from the blocks:**
> **Role:** Act as a senior QA reviewer.
> **Task:** Review the test plan below for coverage gaps and unclear steps.
> **Context:** It's for a checkout flow with guest checkout and discount codes.
> **Format:** List issues in a table — section, issue, severity.
> **Constraints:** Only flag issues; don't rewrite the plan.
> **Input:** [paste plan]

Same request, a night-and-day difference in result — because it's now a spec, not a wish.

### Not a one-shot

Even a well-built prompt often needs a round or two of refinement (see *Prompt Iteration*). The anatomy gives you the parts; iteration tunes them.

## Try it yourself *(optional)*

Take a prompt you dashed off recently and label which blocks it had and which it missed. The missing block is usually exactly what was wrong with the output — which is the whole diagnostic value of knowing the anatomy.

## Common pitfalls

- **Writing only the task**, omitting context, format, or constraints.
- **Mixing instructions and data** with no separation.
- **Cramming multiple unrelated asks** into one prompt.
- **Over-building a trivial prompt.** Match the effort to the task.
- **Leaning on a role for accuracy** it can't provide.

## Key takeaways

- A prompt is a mini-spec; the clearer the spec, the closer the output.
- The core blocks: task, context, input, format, constraints — plus optional role and examples.
- Not every prompt needs every block; complex tasks benefit from more of them.
- Separate the parts, mind the order, and expect to iterate.
