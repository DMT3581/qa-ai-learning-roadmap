## In one line

The model is literal and fills every gap with a guess, so spell out exactly what you want — the task, the format, the scope, the limits — and say what to do rather than what not to do.

## Why this matters for QA

Writing clear constraints for AI is the same skill as writing good acceptance criteria. A vague instruction to a model is like a vague requirement: you can't tell whether the output "passes," because you never defined what passing means. QA engineers are already trained to spot ambiguity in requirements — point that same instinct at your own prompts, and the quality of what you get back jumps. Every piece of vagueness you leave in a prompt is a dice roll on the output.

## Learning objectives

After this chapter you will be able to:

- Turn vague requests into specific, constrained instructions.
- Frame instructions positively (what to do) rather than as prohibitions.
- Specify format, scope, length, and tone so the result is predictable.
- Use hard constraints for genuine non-negotiables.

## Core content

### Instructions vs. constraints

An instruction is what you want done ("summarize this bug report"). A constraint is a boundary on how ("in 3 bullets, focused on reproduction steps, max one sentence each"). Most disappointing outputs are missing constraints, not missing instructions — the model did a version of what you asked, just not the version you had in mind.

### Be specific — the model is literal

Models are radically literal and probabilistic: anything you don't specify, they decide for you, and not always the way you'd choose. "Summarize this" can return three words or three paragraphs. Pin down whatever matters: length, format, focus, tone, audience. The more precisely you describe the outcome you want, the closer you land.

### Say what to do, not what not to do

Positive instructions are more reliable than prohibitions, and there is evidence models follow negative instructions less consistently. Anthropic explicitly recommends telling the model what to do instead of what to avoid. Reframe:

- "Don't be too technical" → "Explain it so a junior manual tester understands."
- "Don't use markdown" → "Write in plain prose paragraphs."
- "Don't be long" → "Keep it under 150 words."

The positive version gives the model a target to aim at instead of a thing to avoid.

### Specify the output format

This is the single most commonly missing constraint. State it plainly: a table with named columns, a numbered list, JSON with a given schema, code in a specific language and style, a maximum length. If you'll reuse the output downstream, the format is not optional.

### Set the scope

Tell the model where to start and stop, and what to include or leave out — positively where you can ("cover only the checkout flow" rather than "ignore everything else"). Bounding the scope keeps the answer on target and stops it wandering into things you didn't ask for.

### Use consistent terminology

If you call something a "test plan," keep calling it that — don't switch to "the document," then "the plan," then "the spec." Inconsistent terms make the model guess whether you mean the same thing, especially in longer instructions.

### Give a fallback

Tell the model what to do when it can't fully comply — for example, "if a requirement is ambiguous, list your assumptions instead of inventing details," or "say 'insufficient information' rather than guessing." This is one of the simplest ways to reduce hallucinations.

### Hard constraints vs. preferences

Match your wording to how firm the rule is. Use absolute language for non-negotiables ("output exactly five cases", "the response must be valid JSON"), and softer wording for preferences. Negative phrasing is fine — and appropriate — for hard boundaries: "must never include real customer data" is a legitimate, important constraint (see Confidentiality and Sensitive Data). The guidance to "prefer positive framing" is about everyday instructions, not absolute safety limits.

> Don't over-constrain trivial tasks. A one-line fix doesn't need a paragraph of rules — match the effort to the stakes.

## Examples

**Weak.** "Review this test plan." → The model rewrites it, or comments on style, or praises it — you don't know which you'll get.

**Strong.** "Review this test plan for missing edge cases and unclear steps. List each issue in a table with columns: location, issue, severity (high/medium/low), suggested fix. Only flag issues — do not rewrite the plan. If a section is fine, skip it."

Specific task, explicit format, clear scope, positive framing, and a fallback for the "fine" case.

## Common pitfalls

- Leaving format, length, or scope unspecified. The biggest source of "not what I wanted."
- Negative-only instructions. Reframe "don't…" as "do…" for everyday guidance.
- Inconsistent terminology. Pick one name per thing and keep it.
- Omitting or burying the actual instruction under context.
- Over-constraining small tasks. Effort should track the stakes.

## Key takeaways

- Specify everything that matters — the model fills unspecified gaps by guessing.
- Prefer positive instructions; reserve negatives for hard boundaries.
- Always state the output format and scope.
- Give a fallback so the model surfaces uncertainty instead of inventing.
