## In one line

When a prompt doesn't give you what you want, don't rewrite it at random — diagnose what's actually wrong, change one thing at a time, and treat it like reproducing a bug.

## Why this matters for QA

First prompts rarely land perfectly, and that is normal — getting good output is a refinement loop, not a one-shot. The good news is that the discipline this takes is one QA already has. Improving a prompt is like narrowing down a defect: you don't change five things and hope, you form a hypothesis about what's wrong, change one variable, and observe the result. Engineers who iterate deliberately get usable results in two or three tries; those who poke at it randomly burn ten attempts and a lot of patience.

## Core content

### Iteration is the normal path

A useful first prompt that needs no improvement is the exception, not the rule. Expect a cycle:

1. **Prompt** — your best attempt (see Prompt Anatomy).
2. **Review** — read the output critically against what you actually wanted.
3. **Diagnose** — name specifically what is wrong.
4. **Refine** — make a targeted change.
5. **Repeat** — until it's good enough.

The step people skip is diagnose. Rewriting before you know what's wrong is just guessing.

### Diagnose, then fix

Match the symptom to the fix instead of reaching for "make it better":

| What's wrong with the output | The targeted fix |
|---|---|
| Too generic or vague | Add context and specifics; say who it's for and why |
| Wrong format | State the exact format — table, list, JSON, max length |
| Missing facts / made things up | Provide the reference material; allow "I don't know" |
| Too long or too short | Specify length explicitly |
| Misunderstood the task | Restate the goal plainly; add a worked example (few-shot) |
| Wrong tone or level | Name the audience and tone ("for a junior tester, neutral") |

### Change one thing at a time

Because models are probabilistic, if you change five things at once and the result improves, you have no idea which change did it — and you can't reproduce it. Isolate variables, exactly as you would when narrowing a flaky test. One hypothesis, one change, one observation.

### Be specific in your feedback

"Make it better" gives the model nothing to act on. Tell it precisely what to change: "Halve the length and focus only on the edge cases" or "Keep the structure but rewrite step 3 to assume the user is already logged in." Specific feedback is the single biggest lever in iteration.

### Two ways to iterate — and when to restart

- **Refine in-chat.** Follow-ups in the same conversation ("shorter, add X") are fast and natural. But watch for drift: if the conversation has stopped converging and you're correcting the same misunderstanding repeatedly, stop.
- **Edit the prompt and re-run fresh.** Change the original prompt and start a clean chat. This is more reproducible and is the right move when the in-chat thread has derailed or when you'll reuse the prompt. (This is the "restart" skill from Prompt vs. Conversation: consolidate what you've learned into one clean prompt.)

### Keep versions, and remember outputs vary

For any prompt you'll reuse, keep a short note of what you changed and why — it lets you revert and combine what worked. Two cautions: models are stochastic, so the same prompt can produce different outputs; don't judge a prompt change on a single run if it matters. And prompt behaviour can shift when the underlying model updates, so a prompt that was perfect last quarter is worth re-checking.

## Examples

**Weak feedback.** "That's not quite right, make it better." → The model guesses what you meant and usually changes the wrong thing.

**Strong iteration (diagnosed).**

- Attempt 1 prompt: "Write test cases for the checkout page."
- Output: generic, happy-path only.
- Diagnosis: too vague, no edge cases, no format.
- Attempt 2 prompt: "Write test cases for a checkout page that takes a discount code and supports guest checkout. Include edge cases: expired code, code at usage limit, empty cart, network timeout mid-payment. Output as a table: case, preconditions, steps, expected result."

One targeted change driven by a clear diagnosis — not a random rewrite.

## Common pitfalls

- "Make it better." Vague feedback wastes a turn; say exactly what to change.
- Changing many things at once. You lose the ability to know what helped.
- Patching a derailed chat forever. If it's not converging, edit the prompt and start fresh.
- Judging on one sample. Outputs vary; for prompts that matter, run a few times.
- Not saving the winner. When a prompt finally works, capture it (see Personal Prompt Library).

## Key takeaways

- Treat prompting as a diagnose-and-refine loop, not a one-shot.
- Name what's specifically wrong before you change anything.
- Change one variable at a time and give specific feedback.
- Refine in-chat for speed, but edit-and-restart when the thread drifts or you'll reuse the prompt.
