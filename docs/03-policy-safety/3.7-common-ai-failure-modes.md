## In one line

AI failures aren't random — they follow recognizable patterns, and once you can name what's happening (it's making things up, it's agreeing to please you, it's drifted off your instructions), you know exactly what to check.

## Why this matters for QA

Recognizing failure patterns is pattern-recognition for defects — your core skill, pointed at the tool. Earlier chapters told you AI can hallucinate, flatter, and miscalculate; this one turns that into a quick field guide. Instead of a vague unease that "something's off," you'll spot *which* failure is happening and reach for the right check immediately. The failures are systematic, rooted in how these models work — which is exactly why they're learnable.

## Core content

A practical grouping of the failures you'll meet in everyday chat use, with the tell and the check for each.

### Truthfulness failures

| Failure | What it looks like | How to detect |
|---|---|---|
| **Hallucination / fabrication** | Confident false facts; invented functions, APIs, or libraries that don't exist | Check specifics against a real source; if you can't verify it, don't rely on it (see *Hallucinations*) |
| **Fabricated citations** | Real-looking references, URLs, or DOIs that don't exist or point elsewhere | Open the source; confirm it exists *and* supports the claim (see *Fact-Checking*) |
| **Confident wrongness** | Fluent, authoritative tone regardless of correctness | Remember tone isn't evidence; verify in proportion to stakes (see *Evaluating AI Outputs*) |
| **Outdated information** | Stale facts, old versions, superseded "best practices" | Check recency; recall the knowledge cutoff (see *Practical Limitations*) |
| **Unfaithfulness to source** | You gave it a document, but it answered from memory instead — or cited your source for something it doesn't say | Confirm the answer is actually grounded in what you provided |

### Instruction and agreement failures

| Failure | What it looks like | How to detect |
|---|---|---|
| **Sycophancy** | Praise and easy agreement; caves when you push back, even if you're wrong | Ask for problems not approval; neutral framing; don't argue it into agreement (see *AI as a Peer Reviewer*) |
| **Instruction attenuation** | Claims it did something it didn't — "I verified", "all cases covered" | Check the work, not the claim — confirm it actually did the thing |
| **Constraint / format drift** | Quietly drops a constraint you set (length, format, scope), often later in a long thread | Compare the output against every constraint you gave |
| **Over-refusal** | Hedges or refuses a perfectly legitimate request | Rephrase and clarify your intent |

### Consistency and reasoning failures

| Failure | What it looks like | How to detect |
|---|---|---|
| **Math / counting / logic errors** | Wrong arithmetic, miscounts, broken multi-step reasoning | Recompute or offload exact work to a tool (see *Practical Limitations*) |
| **Inconsistency** | Same prompt, different answers across runs | Re-run important asks; contradictions are a red flag |
| **Context drift ("lost in conversation")** | Forgets or contradicts earlier turns; quality decays in long threads | Watch for contradictions with earlier context; restart clean (see *Prompt vs. Conversation*) |

### A note on tool and agent failures

When a model is given tools or runs as an agent, new failures appear — wrong tool calls, drifting off the original task, and **prompt injection**, where instructions hidden in content the model reads (a web page, a ticket, a file) get followed as if they were yours. These matter most with browsing and agents; for now, be cautious when you ask AI to act on content that contains its own instructions, and treat that content as data, not commands. (Covered further in the Agents chapters.)

### The one rule that covers them all

Across every failure above, the defense is the same principle: **verify the output, not the claim.** Whatever the model *says* it did or *says* is true, the result has to be checked independently. That isn't distrust — it's the same engineering discipline you already apply to code.

## Examples

**Spot the failure — two cases.**
- AI answers a question about your provided spec but the answer contradicts the spec. → *Unfaithfulness to source.* Check: is it actually grounded in what you gave it?
- You ask "did you cover the lockout case?" and it replies "Yes, fully covered." → *Instruction attenuation.* Check: read the cases — don't take the "yes."

Naming the failure tells you which check to run.

## Common pitfalls

- **Watching only for hallucination.** Sycophancy and drift slip past if that's your only filter.
- **Checking the claim, not the output.** "Did you verify?" → "Yes" proves nothing.
- **Assuming a newer or bigger model removes failures.** These are systematic, not version bugs.
- **Trusting one clean answer.** Inconsistency means a single good run isn't proof.

## Key takeaways

- Failures are patterned and learnable — name the failure to know the check.
- Truthfulness, instruction-following, and consistency are the three families to watch.
- The model claiming it did something is not evidence it did.
- Verify the output independently — the same discipline you apply to code.
