# What Is an AI Agent and How It Works

## In one line

An AI agent is a model given tools and a goal that then works in a loop — deciding its own next step, acting, observing the result, and repeating — which makes it far more capable than a chatbot, and far more consequential when it's wrong.

### Why this matters for QA

Agents are the "does it for you" category — the tools that can actually drive a browser, run your tests, triage failures, or turn a ticket into a pull request, which is exactly where your Layer 2 work (Playwright MCP) is headed. Understanding the concept matters for two reasons. It cuts through the hype: a lot of things marketed as "agents" are just a chatbot with a few buttons. And it tells you why agents raise the stakes — one that takes real actions across many steps can compound small errors into a big mess, so it needs sandboxing, guardrails, and human checkpoints in a way a chatbot never did.

## Core content

### Chatbot vs. agent

A plain model produces text; you read it and act. An **agent** is a model equipped with **tools** and a **goal** that reads from and writes to systems outside itself and **iterates** toward that goal — it doesn't just answer, it *acts*. The defining trait, as Anthropic frames it, is that an agent **decides its own path at runtime** rather than following steps you laid out in advance.

### The task → workflow → agent spectrum

A useful ladder for cutting through the hype:

- **Task** — a single model call (summarize, classify, extract). Bounded and predictable.
- **Workflow** — several model calls along a path *you* define (route → draft → check → send). The control flow is yours; the model just fills each step.
- **Agent** — a model using tools in a loop, **deciding its own trajectory**. You give it a goal, not a script.

Many things sold as "agents" are really tasks or workflows. That's fine — but know which you have, because the risk and the responsibility differ sharply.

### How an agent works — the loop

The engine of an agent is a loop: **perceive** (take in the goal and current state) → **plan/decide** (reason about the next step) → **act** (call a tool — run code, search, click, write a file) → **observe** (read the result) → **repeat**, until the goal is met or it gives up. Its parts are the **model** (the reasoning), **tools** (the hands — see *Agent, Skill, MCP* and *MCP in Practice*), **memory/context** (state carried across steps), and a **goal plus guardrails**. The *Plan → Review → Act* pattern is a disciplined version of this same loop.

### Why autonomy raises the stakes

Agents shine on open-ended problems where you can't hardcode the steps and can't predict how many there'll be. But that autonomy has costs. Because an agent acts over many steps, **small errors compound** across the loop; it costs more (many model calls); and a wrong action can have real effects — it already wrote the file, sent the message, or changed the data. So agents need what a chatbot didn't: **sandboxed testing, guardrails, and human checkpoints** at the consequential steps (see *Human-in-the-Loop Automation* and *AI Does Not Replace Human Judgment*). They're also exposed to **prompt injection** — instructions hidden in content they read and then follow as if they were yours (see *Common AI Failure Modes*). A good rule: the simplest thing that works wins — if you know the exact steps, a workflow or a plain script beats an agent.

### Where it fits for QA

This category is your Layer 2 destination: an agent driving the browser via Playwright MCP to run and triage tests, or a coding agent turning an issue into a pull request. Genuinely powerful — and precisely why the guardrails above matter.

## Try it yourself *(optional)*

Next time something is called an "agent," ask one question: does it decide its own steps toward a goal, or follow a fixed path? The answer tells you whether it's really an agent — and how much oversight it needs before you'd trust it with real actions.

## Common pitfalls

- **Calling everything an "agent."** Most are tasks or workflows.
- **Underestimating compounding errors** across many steps.
- **Running an agent on real systems** without a sandbox and guardrails.
- **No human checkpoint** at consequential or irreversible actions.
- **Forgetting prompt injection** — agents can be hijacked by content they read.
- **Reaching for an agent** when a simple workflow or script would do.

## Key takeaways

- An agent is a model using tools in a loop toward a goal, deciding its own path.
- Task → workflow → agent is a spectrum of autonomy; know which you have.
- The loop is perceive → plan → act → observe → repeat, with model, tools, memory, and guardrails.
- Autonomy compounds errors and takes real actions, so sandbox, guardrail, and keep a human in the loop.
