# Model Context Protocol (MCP) in Practice

## In one line

MCP is the standard "USB-C for AI" that lets an agent plug into real systems — files, repos, a browser via Playwright MCP — and the practical skill is less about wiring it up than about doing it safely, because every server you connect hands the agent real access.

### Why this matters for QA

MCP is the plumbing behind your Layer 2 destination: it's what lets an AI agent actually drive a browser through Playwright MCP to run and explore tests, or reach into your repo and tickets. There are two things to take away — what it unlocks (an AI that can *act* on real systems, not just talk about them), and the safety practices that have to come with it, because connecting an MCP server grants an agent genuine access to your data and tools, and that has already been exploited in the wild. As an ambassador you don't need to build servers, but you do need to understand both the capability and the caution.

## Core content

### What MCP is, in a paragraph

MCP (Model Context Protocol) is an open standard — introduced by Anthropic and now widely adopted across the industry — that gives AI apps one consistent way to connect to external tools and data. Before MCP, every tool needed a custom integration for every model (an "N×M" mess). MCP is the standard connector, often described as "USB-C for AI": a tool exposes its capabilities once, and any model can use them. (The concepts are covered in *Agent, Skill and MCP Fundamentals*; this chapter is the practical view.)

### The pieces

- **Host** — the AI app or agent you use (a desktop assistant, an IDE, a coding agent).
- **Client** — the connector inside the host, one per server.
- **Server** — exposes capabilities from an external system: **tools** (actions the AI can take), **resources** (data/context), and **prompts** (templates). Servers run locally or remotely.

The host asks a server what it offers; those tool descriptions are given to the model; when the agent decides it needs one, it calls it — the action step of the loop from *What Is an AI Agent*.

### What it enables in practice

MCP connects an agent to real systems so it can act, not just answer: your files, a GitHub repo, a Jira board, a database — and, most relevant to you, a **browser via Playwright MCP**. Playwright MCP turns browser automation into tools an AI can call: navigate, click, fill forms, read the *structured* page (the accessibility tree, not raw pixels), run scripts, and capture screenshots. In effect it's an AI that can see and work with the web like a human tester — the heart of AI-driven test automation and exploration, and squarely your Layer 2 pilot territory.

### The catch: connecting a server grants real access

This is the part to internalize. An MCP server gives the agent genuine access to whatever it's connected to — your files, your repo, your browser session. That power is exactly why MCP is useful, and exactly why it's risky. The protocol standardizes the *connection*; it does not make it safe by default, and real exploits have already been demonstrated.

The main risks:

- **Prompt injection through content or tool descriptions.** Hidden instructions can ride in content the server reads — for example, a web page your Playwright agent parses — or even in a tool's own description, telling the agent to do something harmful like read a secrets file and send it to an outside server. The agent may follow it as if it came from you (see *Common AI Failure Modes*).
- **Untrusted or malicious servers, and supply chain.** People install servers without knowing what code runs; typosquatted or poisoned servers can harvest credentials or exfiltrate data.
- **Over-broad permissions ("confused deputy").** A server acting with more privilege than you should have.
- **Data exfiltration** — the common headline outcome of the above.
- **Context bloat** — connecting many servers with long tool descriptions pollutes the agent's context and degrades it (see *Context Engineering*).

### Safe practice

- **Use trusted, official servers**, keep them updated and version-pinned; don't install random ones.
- **Least privilege** — restrict what a server can reach (a single project folder, not your whole disk); don't grant blanket access.
- **Keep local servers local** — bound to localhost, never exposed to the network without protection.
- **Human approval for consequential actions** — read-only context retrieval is far safer than autonomous, state-changing actions; put a checkpoint on anything that writes, sends, or deletes (see *Human-in-the-Loop Automation*).
- **Confidentiality** — for sensitive data, prefer a local server and follow the approved-tool rules (see *Confidentiality and Sensitive Data*).
- **Treat an MCP server like any integration surface** — vet it, and log/monitor what it does.

### Where this leaves a Layer 1 reader

You don't need to wire up MCP servers to benefit from this chapter — that hands-on setup is Layer 2 work. The Layer 1 goal is to understand what MCP unlocks and to carry the safety instincts with you when you or the team start using it for real.

## Try it yourself *(optional)*

If you already use a tool that supports MCP (a coding assistant, say), look at which servers and tools are connected and what each one can access. Ask a simple question: does anything here have more access than it needs? That's the least-privilege lens in action — and often the first thing worth tightening.

## Common pitfalls

- **Installing untrusted or random MCP servers.**
- **Granting blanket access** instead of least privilege.
- **Exposing a local server to the network.**
- **Letting an agent take state-changing actions** with no human checkpoint.
- **Forgetting that content — and tool descriptions — the agent reads can carry injected instructions.**
- **Connecting so many servers the context bloats** and the agent degrades.

## Key takeaways

- MCP is the standard connector that lets agents use external tools and data consistently.
- Host → client → server; servers expose tools, resources, and prompts.
- Playwright MCP is the practical bridge to AI-driven browser testing — your Layer 2 path.
- Connecting a server grants real access; use trusted servers, least privilege, and human checkpoints.
