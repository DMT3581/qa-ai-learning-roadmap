# Plan → Review → Act Workflow

## In one line

Plan → Review → Act is "measure twice, cut once" for AI: the agent proposes a plan, you review and approve it, and only then does it act — so you catch a misunderstanding while it's still cheap to fix, before it's baked into ten executed steps.

### Why this matters for QA

This is test-planning discipline applied to AI. You'd never run a whole suite blind — you plan the coverage, review it, then execute. The same move keeps agentic AI safe and useful: agents are biased toward action and can be confidently wrong, so an agent that starts editing immediately might fix one thing and break three because it didn't grasp the full picture. Inserting a pause between *understanding* and *action* is what prevents "the AI confidently did the wrong thing across ten steps." And it's not abstract — the tools you'll use (Claude Code, Copilot) build this in as an explicit "plan mode."

## Core content

### The pattern

Three phases, with a human checkpoint in the middle:

1. **Plan.** Before touching anything, the agent gathers context and proposes an approach — the steps it will take, the files it will change, the risks. Nothing is executed yet.
2. **Review.** You read the plan, and edit, approve, or reject it. This is the checkpoint. The plan is a document under negotiation until you approve it — you can push back ("that's too invasive, show me another approach") and get a fresh one.
3. **Act.** Once approved, the agent executes the plan, then verifies its work — ideally still pausing for approval on the genuinely consequential steps (see *Human-in-the-Loop Automation*).

It's the disciplined form of the agent loop from *What Is an AI Agent*, with a deliberate pause built in.

### Why it works

- **It catches errors when they're cheap.** A wrong plan reviewed in thirty seconds beats a wrong pile of executed actions you now have to untangle — merge conflicts, data loss, a broken build.
- **Planning surfaces the agent's interpretation.** Reading the plan is how you spot that it misunderstood your intent, *before* it acts on that misunderstanding.
- **The plan is a visible, editable artifact.** Opaque planning breeds distrust; a plan you can read and change lets you add the architectural or domain knowledge the agent lacks.
- **It shifts oversight from every step to the strategy.** This is the elegant part: instead of approving dozens of individual actions (which people tune out — the approval fatigue from *Human-in-the-Loop Automation*), you review the whole approach once, up front, where your judgment matters most.

### When to use it — and when not

Plan → Review → Act is insurance for **complex, multi-step, unfamiliar, or irreversible** work: refactors, migrations, anything that deletes, deploys, or changes shared state. For **trivial, well-scoped** tasks it's overhead — entering plan mode for a typo fix is like writing an architecture document to change a lightbulb. Learning when to plan and when to just act is part of getting fluent with AI (and it echoes the cost-vs-value judgment from *Cost vs Value*).

### It's a general discipline, not just for coding agents

The pattern works for *any* non-trivial AI task, not only agents with file access. Before you let AI run with something that matters, ask for the **approach first**, sanity-check it, then let it proceed. "Outline how you'd do this before doing it" turns a risky one-shot into a reviewable plan. Standing constraints (like the instruction files in *Copilot Tools and File Structure*) also shape what the plan is allowed to include.

## Examples

- **A test refactor:** "Before changing anything, list every test file you'd touch and how, and the risks. Don't edit until I approve." → you catch that it planned to modify a shared fixture, and redirect.
- **Changing shared test data:** plan first, review, then act — never let an agent alter a shared environment blind.
- **A big analysis task:** "Outline your approach and the sources you'll use before you start." → you spot a wrong assumption before it's built into the whole output.

## Try it yourself *(optional)*

Take a non-trivial task you'd normally hand to AI in one shot. Instead, ask it to produce a plan first and stop. Read the plan — did it understand the task the way you meant? Adjust, then let it proceed. The gap you catch at the plan stage is one you'd otherwise have found only after the work was done wrong.

## Common pitfalls

- **Skipping the review** and treating the plan as a formality — that's just delayed rubber-stamping.
- **Planning trivial tasks.** Overhead where a direct action would do.
- **Not reading the plan critically.** The point is to catch the misunderstanding, not to click approve.
- **Approving a plan, then ignoring the consequential steps during execution.** Keep checkpoints on the risky actions.

## Key takeaways

- Plan → Review → Act inserts a human checkpoint between understanding and action.
- It catches misunderstandings and errors while they're cheap, before execution compounds them.
- Review the whole strategy up front — more effective and less fatiguing than approving every step.
- Use it for complex or irreversible work; skip it for trivial tasks; apply it to any non-trivial AI task.
