# Documenting and Sharing Effective Prompts

## In one line

A prompt that reliably works is a small tool your whole team could pick up — but only if you document it for someone who wasn't there when you wrote it, and only if the shared library is curated rather than dumped into.

### Why this matters for QA

This is the point where individual AI skill becomes team capability, and it's squarely part of the ambassador job: developing and documenting reproducible prompting scenarios so colleagues don't each rediscover the same thing by trial and error. The payoffs are real — faster onboarding, consistent output across the team, less duplicated effort. But there's a failure mode worth naming up front: a shared prompt library with no curation becomes a swamp of stale, contradictory, and occasionally unsafe prompts that nobody trusts or uses. Sharing well is a different skill from sharing.

## Core content

### Personal library → team asset

Start from *Personal Prompt Library*: you're already saving what works for you. The team practice is a simple extension — when a prompt consistently delivers good results, it doesn't stay in your private notes; it goes into a shared space where colleagues can find it. Share the **proven** ones and keep experiments personal; a shared library is for things that have earned their place.

### Documenting for someone who wasn't there

This is the core skill, and it's why a shared entry needs more than a personal one. Your own note can be terse because you remember the context; a colleague has none of it. A shareable entry needs:

- **A descriptive name** — searchable in the words a colleague would actually type.
- **The prompt itself**, with clear placeholders (`<USER_STORY>`, `<SANITIZED_CODE>`).
- **When to use it** — and, just as usefully, when *not* to.
- **The inputs** it needs, and any prerequisites.
- **An example of good output**, so people can tell whether theirs went wrong.
- **Owner and last-tested date** — who to ask, and whether it's still current.
- **Known limitations** — where it falls down, so nobody rediscovers that the hard way.

The test: could a colleague use this successfully without asking you anything?

### Where to put it

Pick a place people already work — your VitePress portal, a shared doc, a repo, your tool's built-in prompt library, or as repo files (a Copilot prompt file *is* a shared prompt — see *Copilot Tools and File Structure*). Whatever it is, make it **one** searchable place with a simple category structure. Don't build a bespoke system before you have prompts to put in it.

### Curate, or it becomes a swamp

The most common failure is an unmoderated dump that quickly fills with stale and contradictory entries and loses everyone's trust. Lightweight governance is enough:

- **A light review** before something is added — does it work, is it documented, is it safe?
- **Naming conventions** so search works.
- **Clear ownership** for each entry.
- **Periodic cleanup** (say quarterly): retire prompts nobody uses, consolidate near-duplicates, and re-test the important ones — remember prompts drift when models update (see *Model Versions*).
- **Deprecate rather than delete**, so people aren't confused by a prompt that vanished.

### Build good habits into the prompts

A subtle, high-value benefit: the prompts themselves can carry your standards. Include the sanitization reminder ("replace real customer data with placeholders"), the verification step ("check outputs against the requirement"), and a note where human review is mandatory. Done this way, the library isn't just a productivity tool — it quietly nudges the whole team toward safe, responsible use (see *Confidentiality and Sensitive Data* and *Evaluating AI Outputs*). Never store secrets or real client data in a shared prompt.

### Make sharing normal

Not everyone shares by default — some hesitate, some just forget. As an ambassador you can lower the barrier: make submitting easy, credit contributors, and demonstrate the library in sessions so people see it used. A library nobody knows about is the same as no library (see *Building Team Knowledge Around AI Usage*).

## Examples

- **A `generate-test-cases` entry** — the prompt with placeholders for story and rules, a note that it needs the acceptance criteria, an example output table, your name, a tested date, and a caveat that it misses domain-specific business rules.
- **A bug-report polisher** shared with the whole QA team → reports become more consistent across people, which developers notice.
- **A prompt with the sanitization reminder built in** → colleagues who never read the confidentiality chapter still get nudged toward the right habit.

## Try it yourself *(optional)*

Take your single best prompt, write it up with the fields above, and hand it to a colleague without explaining it. Watch where they get stuck — that gap is exactly the documentation you were missing, and fixing it is what makes the entry genuinely shareable.

## Common pitfalls

- **Sharing a bare prompt with no context.** Others can't tell when or how to use it.
- **Sharing unproven experiments.** Keep those personal.
- **No curation.** The library turns into a swamp and loses trust.
- **Scattering prompts** across chats, docs, and DMs instead of one home.
- **Leaving real data or secrets** in a shared prompt.
- **Building the library but never promoting it.** Unknown equals unused.

## Key takeaways

- Share proven prompts; keep experiments personal.
- Document for someone who wasn't there: name, placeholders, when to use, example output, owner, date, limitations.
- Keep one searchable home and curate it — review, own, clean up, deprecate.
- Embed sanitization and verification into prompts so the library reinforces safe habits.
