## In one line

Traditional software does exactly what a human told it to do, step by step; AI learns patterns from examples and produces the most likely answer — which means it is powerful, flexible and sometimes confidently wrong.

## Why this matters for QA

Every QA engineer has an intuition built on traditional software: the same input gives the same output, behaviour is defined by a specification, and a bug is a traceable mistake in the logic. AI breaks all three of those assumptions. The same prompt can give different answers, there is no exact specification of "correct," and a wrong answer is not a line of code you can point to. Understanding this difference is the foundation for everything else in this program: it shapes how much you can trust an AI's output, how you use it day to day, and — later — how you test AI-powered features that no longer behave deterministically.

## Learning objectives

After this chapter you will be able to:

- Explain in plain language what "AI" means in the tools you use at work.
- Describe the core differences between traditional, rules-based software and AI.
- Explain why AI produces probable answers rather than guaranteed ones.
- Recognize what this changes about how you use and trust AI.

## Core content

### What "AI" means in practice today

"Artificial intelligence" is a broad umbrella for any system that performs tasks we associate with human intelligence. In your daily work, though, "AI" almost always means a narrower thing: systems built with machine learning (ML) — and especially large language models (LLMs) like the ones behind ChatGPT, Claude, Gemini, and Copilot. The defining trait of these systems is that they learn from data instead of following hand-written rules. So the useful contrast for us is not "robots vs. humans" but rules-based software vs. systems that learn.

### How traditional software works

In traditional software, a person writes explicit rules and logic. The developer anticipates the cases, encodes them as instructions, and the computer follows them exactly. Because every decision traces back to a written rule, traditional software is:

- **Deterministic** — the same input always produces the same output.
- **Transparent** — you can read the logic and explain every result.
- **Predictable** — it does precisely what was coded, no more, no less.
- **Rigid** — it only handles cases the developer thought of; anything else is unhandled.

A bank's interest calculator is a good example: a fixed formula, the same result every time, fully auditable.

### How AI works

In an AI system, nobody writes the rules. Instead, the model is trained on large amounts of data, and it derives statistical patterns from that data on its own. When you give it new input, it produces the output that those patterns make most likely. This makes AI:

- **Probabilistic** — it predicts a likely answer, so the same input can yield different outputs.
- **Pattern-based** — it generalizes to new situations it was never explicitly programmed for.
- **Opaque** — the "reasoning" lives in millions of learned parameters, not readable rules, so results are hard to fully explain.
- **Flexible but fallible** — it handles messy, open-ended problems well, but can produce fluent answers that are simply wrong.

The classic illustration is a spam filter. The traditional version is a list of rules a developer maintains ("if the email says 'win money', mark as spam"), updated by hand whenever spammers change tactics. The ML version is trained on thousands of example emails labelled "spam" or "not spam," and learns the patterns itself — adapting to new spam it has never seen, but occasionally getting it wrong.

### Side-by-side

| | Traditional software | AI (machine learning / LLMs) |
|---|---|---|
| Built by | Writing explicit rules | Training on data |
| Same input, same output? | Yes (deterministic) | Not guaranteed (probabilistic) |
| Handles new/unseen cases | Only if coded for | Generalizes, but may be wrong |
| Can you explain each result? | Yes, trace the logic | Hard — patterns, not rules |
| Typical failure | Logic error you can debug | Plausible-sounding wrong answer |
| Best for | Well-defined, stable tasks | Fuzzy, pattern-rich, open-ended tasks |

### So what changes for you

Because AI gives likely answers rather than guaranteed ones, you cannot trust it the way you trust a calculator. Fluent, confident output is not the same as correct output. The right mental model is a knowledgeable but fallible assistant: fast and broadly capable, but something whose work you verify before you rely on it. It is neither magic nor "just autocomplete" — it is sophisticated pattern prediction, and knowing that is what lets you use it well.

## Examples

**A QA framing — the test oracle.** With a traditional feature, you usually have a deterministic expected result, so you can assert on it directly: input 2 + 2, expect exactly 4. With an AI feature (say, a model that summarizes a bug report), there is no single correct output — many summaries are valid. Testing shifts from "does it equal the expected value?" to checking properties (is it factual? within length? free of leaked data?), accepting ranges of acceptable answers, and using human review. Same testing instinct, different oracle.

**Everyday framing.** Ask an AI tool the same question twice and you may get two differently worded answers — both reasonable. That is not a bug; it is the probabilistic nature of the system. Traditional software giving two different answers to the same input would be a bug.

## Common pitfalls

- Treating AI like deterministic software. "It got it right once, so it always will." Not guaranteed — re-checking matters.
- Assuming it "knows" facts like a database. It predicts plausible text; plausible is not the same as true (see hallucinations).
- Trusting fluency. A confident, well-written answer can still be wrong. Tone is not evidence.
- Dismissing it as a toy. Underestimating AI is as unhelpful as over-trusting it; the goal is calibrated use.

## Key takeaways

- Traditional software follows hand-written rules; AI learns patterns from data.
- AI is probabilistic and opaque, so the same input can give different — and sometimes wrong — answers.
- Fluent output is not proof of correctness; verify before relying on it.
- This shift is the foundation for how you use, trust, and eventually test AI.
