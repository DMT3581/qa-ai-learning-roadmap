## In one line

A context window is how much text (in tokens) a model can hold in mind at once — its working memory for this conversation — and anything outside it, including everything from past chats, simply doesn't exist to the model.

## Why this matters for QA

This explains two things you'll hit constantly: why AI "forgets" what you told it earlier in a long conversation, and why a huge pasted document loses its middle. The context window is the model's working memory — finite, and crucially *not* persistent between sessions. Once you know it works like RAM, not a hard drive, you stop expecting it to remember yesterday's chat and start deliberately re-supplying what it needs.

## Core content

### What a context window is

The context window is the finite amount of text — measured in tokens (see *Tokens Explained*) — that a model can actively consider at once. Everything in the current prompt and conversation, *plus* the response it generates, has to fit inside it. It's the model's working memory for this interaction: what it can "see" right now. If something falls outside the window, the model effectively can't see it — the same rule as in *Context Engineering*: if it's not in the window, it doesn't exist.

Windows have grown fast — from a few thousand tokens a couple of years ago to hundreds of thousands, and a million-plus in some models — and they vary by model.

### It's RAM, not a hard drive

The key mental model: a context window is like working memory (RAM), not persistent storage. Three consequences follow:

- **Volatile.** Everything in it vanishes when the session ends — a preference you stated in turn 1, a constraint you set in turn 3. By default the model remembers nothing across separate conversations; every new chat starts blank.
- **Degrades under load** (next section).
- **Costly.** Every message re-processes the whole window, so a bloated context is slower and more expensive on every single turn.

So AI has no persistent memory by default. "Memory" features in some apps aren't the model truly remembering — they're the app quietly re-supplying context each time.

### Bigger isn't automatically better: "lost in the middle"

Within the window, attention isn't uniform. A well-documented effect — "lost in the middle" (Liu et al.) — shows models recall information at the **beginning and end** of a long context far better than in the **middle**, where accuracy can drop sharply; and quality tends to degrade as the window fills, not only at the limit. A model can technically read a million tokens and still reason poorly over a dump of them. So a bigger window helps, but it's not a licence to paste everything (see *Context Engineering*).

### What happens when you exceed it

The request either fails or your input gets **truncated** — often silently. In a long chat, the earliest turns fall out of view and the model "forgets" them; in a big pasted document, part simply gets dropped and never considered.

### Practical responses

- **Re-supply what matters.** Don't assume it remembers earlier chats or even earlier turns; put the relevant context back in.
- **Summarize and restart** long conversations instead of letting them balloon (see *Prompt vs. Conversation*).
- **Place key info where it's noticed** — near the start or end, not buried in the middle.
- **Chunk big inputs**, or check they fit (see *Tokens Explained* and *Tool Limitations*).

## Examples

- **A long test spec pasted whole** → the middle is under-attended or truncated; chunk it, or put the critical parts at the edges.
- **A long debugging chat** → it "forgets" the constraint you set twenty messages ago; restate it, or restart with a summary.
- **Expecting it to recall last week's session** → it won't; re-supply the context.

## Try it yourself *(optional)*

In a long conversation, ask the model to repeat a specific detail you gave near the very start. Notice whether it still has it — that's a quick, concrete feel for where the window's edges are.

## Common pitfalls

- **Expecting cross-session — or even long-chat — memory.**
- **Treating "memory" features as the model truly remembering.** It's re-supplied context.
- **Dumping everything into a big window** and assuming it's all used well.
- **Burying the key detail in the middle** of a long context.
- **Not noticing a big input was silently truncated.**

## Key takeaways

- A context window is the model's working memory, measured in tokens; input and output share it.
- It's volatile like RAM — no persistent memory across sessions by default.
- "Lost in the middle" means a bigger window isn't a free place to dump everything.
- Re-supply context, summarize and restart, and place important information at the edges.
