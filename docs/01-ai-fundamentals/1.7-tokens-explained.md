## In one line

A token is the chunk of text an LLM actually reads and writes — usually a word or word-piece, not a letter — and this one design choice explains AI's costs, its context limits, and why it fumbles counting letters.

## Why this matters for QA

Tokens are the unit everything is measured and billed in, and they quietly explain a cluster of behaviours you'll run into. Context limits are counted in tokens (so a big input can silently overflow), cost is per token, and the reason AI miscounts letters — and sometimes botches arithmetic — is that it never sees letters at all, only these chunks. Knowing this stops you from trusting AI to do the things tokenization makes it bad at, and helps you keep prompts efficient.

## Learning objectives

After this chapter you will be able to:

- Explain what a token is and why LLMs use them.
- Convert roughly between tokens and words.
- Connect tokenization to cost, context limits, and character-level failures.

## Core content

### What a token is

A token is the smallest unit an LLM reads and writes — not a word and not a letter, but a chunk somewhere in between. Common words are usually a single token; longer or rarer words get split into pieces ("strawberry" → "straw" + "berry"); spaces and punctuation count too. Each token is turned into a number the model processes. A rough rule for English: about three-quarters of a word per token (~4 characters), so roughly 1.3 tokens per word.

### Why chunks, not words or letters

Using whole words would need an impossibly large vocabulary and still fail on words it had never seen; using single letters would make every sequence painfully long. Subword tokenization (commonly Byte-Pair Encoding) is the compromise: a manageable vocabulary that can still represent *any* text — familiar words stay whole, and unfamiliar ones are built from pieces.

### Tokens are the unit of cost and limits

Both what you send and what you get back are counted in tokens. API pricing is per token, and a context window is measured in tokens — a "128K context" means 128,000 tokens (roughly 96,000 English words), not words (see *Context Windows and Memory Limitations*). Shorter, clearer prompts use fewer tokens: cheaper, and more room left over (see *Token-Efficient Prompting Strategies*).

### Why this explains the letter-counting fumble

Because the model sees tokens, not letters, it has no direct view of the characters inside a word. That's why "how many r's in strawberry?" trips it up — it sees chunks like "straw" + "berry", and the individual letters are hidden. Modern models often get it right now (through targeted training, spelling the word out step by step, or calling a tool), but the underlying limitation is still there for subtler character tasks. The same chunking makes exact arithmetic shaky, because numbers tokenize inconsistently. The fix is to offload these to a tool — have it write code to count or compute rather than answer from its head (see *Practical Limitations of AI*).

### Not all text tokenizes equally

Tokenizers are trained mostly on English, so other languages — and code — often need more tokens to express the same meaning. Working in Romanian, the same text will typically cost more tokens (and more context) than its English equivalent. Worth knowing when you're near a limit or watching cost.

## Examples

- **Pasting a long spec** that overflows the token limit → part is silently dropped, and the analysis misses it (see *Context Windows* and *Tool Limitations*).
- **"How many test cases are in this list?"** → don't trust the count; the model isn't reliably counting items or characters. Count yourself, or have it write a script (see *Practical Limitations of AI*).
- **A Romanian prompt** uses more tokens than the same content in English.

## Try it yourself *(optional)*

Paste a paragraph into a tokenizer tool (such as OpenAI's tiktoken viewer) and watch how your words split into tokens. Then try the same paragraph in Romanian and in English and compare the counts — the difference is the "not all text tokenizes equally" point, made concrete.

## Common pitfalls

- **Thinking in words** when limits and costs are counted in tokens.
- **Trusting the model to count** letters, characters, or items — tokenization hides them.
- **Assuming non-English text costs the same** as English. It usually costs more.
- **Overflowing the context** and not realizing your input got truncated.

## Key takeaways

- A token is a subword chunk — the unit an LLM actually reads and writes.
- Cost and context limits are measured in tokens, not words (~1.3 tokens per English word).
- The model can't see letters inside tokens, which is why counting and spelling can fail.
- Other languages and code use more tokens per meaning — relevant to cost and limits.
