## In one line

At each step an LLM doesn't pick *the* next word — it samples from a probability distribution over many possible words, which is why the same prompt gives different answers and why you can never treat it like deterministic software.

## Why this matters for QA

This is the single fact that most breaks a tester's instincts. In the software you know, the same input gives the same output, and any difference is a bug. With an LLM, variation is *expected behaviour*, not a defect — and that changes two things: how you use it (don't count on reproducibility; verify; record the model when it matters) and how you test AI-powered features (you can't assert exact-output equality; you check properties and acceptable ranges instead).

## Learning objectives

After this chapter you will be able to:

- Explain why LLM output is probabilistic.
- Understand, at a high level, what temperature does.
- Recognize that even "deterministic" settings aren't perfectly reproducible.
- Apply the consequences: expect variation, verify, and test with tolerance.

## Core content

### Why it's probabilistic

Recall from *What Is a Large Language Model* that a model generates one token at a time. At each step it doesn't produce a single answer — it produces a **probability distribution** over every possible next token (this word 60% likely, that one 25%, and so on). Then a **sampler** picks one from that distribution. Because there's a selection — often with a dash of randomness — among several plausible options, the output isn't fixed. Run the same prompt twice and you'll often get two different, both-reasonable answers. (Ask for a haiku twice and you'll get two different haikus.)

### The temperature dial

Most tools expose, or set internally, a **temperature** that controls how adventurous the sampling is. Low temperature → the model sticks to the most likely tokens (more focused, more repeatable); high temperature → it ventures into less likely ones (more varied, more creative). At temperature 0 it simply takes the single most likely token each time ("greedy" decoding). So variability can be dialled up or down — though, as below, not reliably to zero.

### It's a feature, not a bug

This variability is *why* LLM text reads naturally instead of robotically. A model that always produced the single most probable continuation would be repetitive and bland; sampling is what gives it range and fluency. Contrast this with traditional software, which is deterministic by design (see *What AI Is*): there, two different outputs for one input *is* a bug; here it's normal.

### Even "deterministic" isn't guaranteed

Worth knowing: even at temperature 0, providers don't promise identical results — both Anthropic and Google note that a small amount of variation is still possible. The reasons are low-level: floating-point arithmetic, how work is parallelized on GPUs, and how your request happens to be batched with others. Larger models tend to vary more. The practical takeaway is simple — never assume an LLM is perfectly reproducible.

### What this means in practice

- **Expect variation.** Different answers to the same prompt is normal, not a fault.
- **Don't rely on reproducibility.** If a result must be repeatable or shared, record the model and settings — and know it may still drift (see *Model Versions and Why They Matter*).
- **Verify and cross-check.** Running a prompt again and comparing is a legitimate check; big disagreement is a red flag (see *Evaluating AI Outputs*).
- **For consistency, constrain.** Specify the exact output format and lower the randomness where you can — then still verify.
- **Judge across several runs, not one.** For anything you're measuring, a single run isn't evidence (see *Measuring Personal Productivity Gains*).

### The QA-testing angle

This reshapes how you test AI-powered features. You usually can't assert that the output *equals* an expected string, because it legitimately varies. Instead you test **properties** (is it valid, factual, within length, correctly formatted?), **acceptable ranges**, and **semantic equivalence** — and you run it multiple times. The deterministic exact-match oracle simply doesn't apply.

## Try it yourself *(optional)*

Ask the same open-ended question three times and compare the answers. Notice how the wording changes but the gist usually holds — that's sampling in action, and it's exactly why "does it equal X exactly?" is the wrong test for an LLM.

## Common pitfalls

- **Treating variation as a bug.** It's expected behaviour.
- **Assuming a good answer will reproduce identically.** It may not.
- **Assuming temperature 0 means perfect reproducibility.** It doesn't.
- **Asserting exact-output equality** when testing AI features.
- **Judging a prompt or model on a single run.**

## Key takeaways

- LLMs pick each token by sampling from a probability distribution, so outputs vary.
- Temperature tunes the randomness; low is focused, high is creative, zero is greedy.
- Even at temperature 0, hardware and batching prevent guaranteed reproducibility.
- Expect variation: verify, record the model when it matters, and test with tolerance, not exact match.
