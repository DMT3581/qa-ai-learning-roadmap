# AI for Test Design and Test Case Generation

## In one line

AI can turn a requirement into a thorough set of test cases in seconds — often including edge cases you'd overlook — but only when you bring the test-design thinking and review what comes back.

### Why this matters for QA

This is the most immediate, visible win AI offers a QA engineer. Translating requirements into structured test scenarios is slow, repetitive work, and AI does a strong first pass in seconds — frequently surfacing edge cases like rate limiting, boundary conditions, or injection attempts that a tired human misses. In studies comparing AI-generated and manually written test cases, the AI output has scored comparably on correctness, consistency, and completeness. But the headline finding everywhere is the same: it works *with* a skilled tester, not instead of one. Garbage in, garbage out — vague requirements produce shallow tests. The value comes from you supplying good context, naming the right technique, and reviewing critically.

## Core content

### What AI is good and bad at here

**Strong at:**

- **Speed** — a first draft of a test suite in seconds instead of an hour.
- **Breadth** — surfacing valid, boundary, and negative cases, including edge cases you might not think of.
- **Format** — producing structured output (tables, Gherkin, automation skeletons) and converting between them.
- **Beating the blank page** — giving you something to react to and refine.

**Weak or risky at:**

- **Vague requirements** — with thin input it produces shallow, generic, happy-path tests (GIGO).
- **Redundancy and inconsistent coverage** — duplicate cases, uneven depth.
- **Nuanced business rules** — it cannot know domain constraints you didn't tell it, and will quietly skip or invent them.
- **False completeness** — fifty cases *look* thorough; looking thorough is not the same as covering the risk.
- **Inconsistency** — the same prompt yields different tests each run.

### The golden rule

**AI amplifies test design; it does not replace it.** You still own the parts that carry the skill: *what* to test (risk-based judgment), *which* techniques apply, what "adequate coverage" means here, and whether each case reflects a real business rule. Hand it those decisions and it does the mechanical translation and breadth brilliantly. Skip them, and you get a fast pile of shallow tests.

### Drive it with a test-design technique

The single biggest quality lever is naming the technique instead of saying "write tests." Point the model at the right tool for the input:

- **Equivalence partitioning + boundary value analysis** — for fields with ranges and limits.
- **Decision tables** — for combinations of conditions and rules.
- **State-transition** — for workflows and state machines.
- **Negative and edge cases** — invalid input, error paths, limits, and security (injection, auth bypass).
- **Pairwise / combinatorial** — to tame an explosion of combinations.

Naming the technique focuses the model and sharply improves coverage.

### Give it the right context

Apply *Context Engineering* here directly. Supply:

- the **requirement / user story / acceptance criteria**,
- the **constraints** it can't infer (validation rules, business rules, limits),
- the **output format** you want (table columns, Gherkin, Playwright + TypeScript),
- a short **example** of your house style.

And first, the confidentiality gate from *Confidentiality and Sensitive Data*: sanitize before pasting, and for client code use an approved enterprise tool, never a consumer one.

### A practical workflow

1. **Prompt** with context + technique + format.
2. **Generate** the first pass.
3. **Review critically** — this is where your value is: remove duplicates, check coverage against the actual risk, confirm each case maps to a real requirement (cut any invented behaviour), fix weak assertions, and add the domain-specific cases the AI couldn't know.
4. **Iterate** the prompt for the gaps you found (see *Prompt Iteration*).
5. **Merge / automate** only after review.

This *generate → validate → merge* loop is the standard precisely because it avoids blind trust (see *Evaluating AI Outputs* and *AI Does Not Replace Human Judgment*).

### Copy-paste prompt patterns

**From a user story, with a technique:**
> "You are helping design tests. For this user story, generate test cases using equivalence partitioning and boundary value analysis. Story: *As a user, I can set a password between 8 and 64 characters.* Output a table: ID, technique, input, precondition, steps, expected result. Include valid, boundary, and invalid partitions. List any assumptions separately."

**Negative and security edge cases:**
> "For the login form below, generate only negative and edge-case tests: invalid credentials, lockout, malformed input, and common security cases (SQL injection, XSS in inputs). Output as a table with expected result. Do not include happy-path cases. [paste sanitized field list]"

**Bridge to automation:**
> "Convert these reviewed test cases into Playwright + TypeScript test skeletons, following this existing test as the style pattern: [short example]. Leave assertions as TODOs where the expected value depends on environment data."

## Try it yourself *(Lab)*

**Setup:** the Conduit (RealWorld) app and the "Sign up" feature (fields: username, email, password; email must be valid and unique).

**Steps:**
1. Write a prompt that includes the feature's rules, asks for **boundary value + negative** cases, and specifies a table format.
2. Generate the first pass.
3. Review: mark every case as *keep*, *duplicate*, *invented* (behaviour not in the spec), or *fix*. Then write down **two cases the AI missed** that you'd add from your own knowledge of sign-up flows.

**Expected outcome:** a reviewed, de-duplicated test set — plus your short list of what the AI got wrong or missed. The learning is in the review, not the generation.

**Stretch:** regenerate the same prompt and compare the two outputs. Note what changed — that variability is the stochasticity from *Practical Limitations* in action.

## Examples

**Weak.** "Write test cases for the login page." → A dozen generic happy-path cases, no edge cases, no format you can use.

**Strong.** A prompt with the login rules, the instruction to apply boundary + negative + security techniques, a table format, and a request to list assumptions → structured cases including lockout, malformed email, and injection attempts — that you then review and prune.

> **What AI typically misses here:** the *business-specific* rule. If your sign-up silently lowercases emails, or blocks disposable-email domains, the AI won't know unless you tell it — and it may confidently produce tests that assume the opposite. Those gaps are exactly where your expertise earns its place.

## Common pitfalls

- **"Write tests for X" with no context or technique.** You'll get generic happy-path output.
- **Accepting the pile without review.** Duplicates, invented behaviour, and missing domain cases hide in volume.
- **Mistaking quantity for coverage.** Count is not risk coverage.
- **Pasting proprietary code into consumer tools.** Use approved enterprise tools (see *Confidentiality*).
- **Trusting it on nuanced business rules.** It can't know what you didn't tell it.
- **Expecting identical output each run.** It's probabilistic — pin down what matters in the prompt.

## Key takeaways

- AI gives you a fast, broad first draft of test cases — review is where your value lives.
- Name the test-design technique; it's the biggest lever on quality.
- Supply context and format, and sanitize first.
- Use the generate → validate → merge loop; never merge unreviewed.
